# 코드 작성 후 워크플로우

코드를 작성하거나 수정한 뒤, 반드시 아래 단계를 순서대로 수행한다.

## 1단계: 린터 실행

변경한 패키지에 대해 린트를 실행하고, 에러가 있으면 모두 수정한다.

```bash
# 변경 패키지에 맞게 실행
pnpm --filter @game-hub/server lint    # apps/server 변경 시
pnpm --filter web lint                 # apps/web 변경 시
pnpm --filter @game-hub/shared-types lint  # packages/shared-types 변경 시

# 여러 패키지 변경 시 전체 실행
pnpm lint
```

- 린트 에러가 발생하면 즉시 수정 후 다시 린트를 통과시킨다.
- 자동 수정 가능한 항목은 `--fix` 플래그를 활용할 수 있다.

## 2단계: 테스트 코드 작성 및 실행

변경한 로직에 대한 테스트를 작성하고 실행한다.

### 테스트 작성 기준

- 새로운 기능을 추가한 경우: 해당 기능의 핵심 동작을 검증하는 테스트 작성
- 버그를 수정한 경우: 해당 버그를 재현하는 테스트 작성
- 리팩토링한 경우: 기존 테스트가 통과하는지 확인, 필요 시 테스트 수정
- 타입만 변경한 경우: 타입 가드/유틸리티가 있으면 테스트, 순수 타입 정의만이면 생략 가능

### 테스트 실행

```bash
# 서버 테스트 (커버리지 포함)
pnpm --filter @game-hub/server test:coverage

# 웹 테스트 (커버리지 포함)
pnpm --filter web test:coverage

# 특정 파일만 테스트 (커버리지 없이)
pnpm --filter @game-hub/server test -- <파일패턴>
pnpm --filter web test -- <파일패턴>
```

- 테스트가 실패하면 코드 또는 테스트를 수정하여 모두 통과시킨다.
- 커버리지가 기준(80%) 미달 시 실패 처리된다. 테스트를 보강하여 통과시킨다.
- 커버리지 측정 대상은 각 패키지의 `vitest.config`에 정의되어 있으며, 설정/래퍼/UI 컴포넌트 등은 제외된다. 상세는 `test-code.md`의 "커버리지" 섹션 참고.
- 테스트 작성 시 `test-code.md` 규칙을 따른다.

## 3단계: 빌드 검증

전체 빌드가 성공하는지 확인한다. 린트는 코드 스타일만 검사하므로, 타입 에러와 번들링 문제는 빌드에서 잡아야 한다.
특히 `shared-types` 변경 시 의존 패키지(web, server) 빌드까지 반드시 확인한다.

```bash
pnpm build
```

- 빌드 에러가 발생하면 수정 후 1단계부터 다시 수행한다.

## 체크리스트

코드 작성 완료 후 아래를 모두 확인한다:

1. [ ] 린터 통과 (에러 0개)
2. [ ] 변경 로직에 대한 테스트 작성 완료
3. [ ] 모든 테스트 통과 + 커버리지 기준(80%) 충족
4. [ ] 빌드 성공 (`pnpm build`)
